Public-repository security boundary

Type: reference and operating policy · Authority: verified I1A implementation plus repository security policy · Status: policy and enforcement active; I1A complete / PASS at recorded state · Updated: 2026-07-21

Assume public and permanent. Source, documentation, tests, fixtures, examples, commits, branches, pull-request diffs, CI configuration and output, caches, generated artifacts, release artifacts, screenshots, and issue attachments can all disclose material. Do not put secret or sensitive values in any of them.

Standing boundary

  • Never commit credentials, tokens, API secrets, service-role keys, private keys, signing material, recovery values, private endpoints, or sensitive environment configuration.
  • Never use production or personal data in fixtures, seed data, screenshots, reports, examples, logs, telemetry, crash output, or documentation. Use synthetic data.
  • Client-side configuration is publishable only after explicit classification, least-privilege review, and server-side authorization review. A client location does not make a value safe. Supabase service-role keys and third-party provider credentials remain server-side secrets.
  • Use ignored local configuration and protected CI secret storage. Check in only redacted templates. Ensure errors, command output, snapshots, test reports, and generated bundles do not echo private values.
  • Do not inspect, copy, quote, or document a suspected value during review. Preserve redacted path/category/fingerprint evidence only.

Required review surface

The implemented scanner examines publishable working-tree paths/content, all staged index paths/blobs, all reachable Git blob/commit/tag/tree objects, Git ref names, and requested Android APK path/archive/entry metadata and content. This includes frozen legacy and dormant iOS through repository enumeration rather than path exemption. Every later implementation gate that can affect configuration, integrations, CI, packaging, signing, classifications, or release must repeat the relevant redacted review. Scanner classifications use exact-value digests and opaque locators; output must never reproduce the matched value.

Suspected disclosure response

  1. Stop publication, copying, CI execution, or release of the affected material when doing so prevents further exposure.
  2. Escalate privately; never paste the suspected value into an issue, chat, command, commit message, or durable document.
  3. Revoke or rotate at the provider first. Deletion from the current tree is not remediation for a disclosed credential.
  4. Coordinate any history rewrite, remote-log/artifact cleanup, provider change, or signing response under separate explicit authorization.
  5. Re-run redacted review across the working tree, history, remote CI surfaces, caches, and artifacts before claiming resolution.
