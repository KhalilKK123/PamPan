Assisted manual TalkBack checklist

Record device/emulator model, Android/API level, system build, TalkBack version when available, artifact/source commit, assessor, date, and PASS/FAIL without sensitive data. The assessment must use the same executable candidate as the accepted CI run.

  1. Install or launch the candidate while network access is unavailable; confirm the deterministic foundation screen opens without account, service, permission, or network prompts.
  2. Enable TalkBack and launch PamPan from the launcher. Confirm the application can be entered and the screen receives accessibility focus.
  3. Swipe forward through the screen. Confirm the visible title “PamPan,” status “Native foundation,” and message “Your pantry, thoughtfully managed.” are announced accurately in logical top-to-bottom order, without an unexpected focus trap or empty/unlabeled focus target.
  4. Swipe backward through the same content. Confirm navigation reverses predictably, focus remains visible, and the user can leave the application using standard system navigation.
  5. Confirm there are no unlabeled actionable controls, inaccessible required actions, unexpected permission dialogs, or network/service error surfaces. The current foundation screen intentionally has no product controls.
  6. Record each item as PASS or FAIL plus concise observations. A partial, inferred, screenshot-only, automated semantics-only, or unobserved result does not satisfy the manual gate.

  • final documentation validation and git diff --check pass without a post-gate behavior-affecting change.

Only then may the larger PAMPAN-NATIVE-REWRITE / I1 be recorded complete / PASS. I1A retains its separate completed result regardless of I1B outcome.

Narrow-remediation boundary

Remediation may touch only the smallest CI/emulator/test harness or Android accessibility semantics necessary to resolve observed I1B evidence failure. It must not introduce product functionality or broaden architecture. Any behavior-affecting or executable configuration change invalidates earlier CI, runtime, TalkBack, verification, and review results; create a new commit/state and restart every I1B gate. A fix requiring a dependency/toolchain upgrade, new permission, service/network behavior, signing material, iOS work, legacy change, or broader UI redesign requires fresh approval.

Stop conditions

  • A suspected secret, credential, private endpoint, signing material, sensitive user data, or unredacted disclosure: stop publication and CI where practical, escalate privately, rotate/revoke externally, and re-verify with redacted evidence.
  • The remote SHA or workflow head_sha does not equal the candidate, the relevant run cannot be identified, or a required job is skipped/cancelled/fails: do not infer PASS.
  • Emulator boot, offline boundary, connected tests, or any manual TalkBack checklist item fails or cannot be observed: keep I1B and larger I1 BLOCKED pending narrow remediation or the missing human/external evidence.
  • A requested fix crosses the remediation boundary or would activate I2–I9, iOS, services, signing, migration, or product behavior: stop for fresh approval.
