Target architecture

Android app (Compose) ── Supabase client APIs ──┬─ Postgres + RLS
                                                ├─ Auth / Storage / Realtime
                                                └─ Edge Functions (narrow only)

non-executable assets: schemas | API/RLS contracts | synthetic fixtures | test vectors
dormant preserved code: apps/ios/ (no active implementation or gate)
  • The Android application owns its UI, navigation, local persistence, state, domain mapping, accessibility behavior, and platform tests. Contracts are proven with Android tests over synthetic fixtures and test vectors, not a shared runtime layer.
  • Direct client access is limited to authenticated, RLS-authorized ordinary operations. Policies are part of the contract and must be tested with at least two principals.
  • Functions receive only the smallest necessary inputs, authenticate/authorize independently, validate schema, provide idempotency where applicable, and return typed public errors. They are mandatory for secrets, privileged cross-user work, billable providers, coordination, push dispatch, and webhooks.
  • Introduce a separate backend only after measured latency, reliability, security, operational, or coordination evidence shows that Supabase primitives and narrow functions are insufficient; record the evidence and obtain approval.

Security, privacy, and data design

  • This repository is public. Treat source, documentation, tests, fixtures, examples, Git history, CI configuration/output, caches, generated/release artifacts, screenshots, local configuration intended for commit, and signing material as potentially public and permanent. Follow the public-repository security boundary.
  • No secrets in source, fixtures, logs, CI output, generated app bundles, or documentation. Client configuration contains only publishable values; service-role keys and third-party credentials remain server-side secret material.
  • Use environment-specific configuration with explicit development/staging/production separation, checked-in redacted templates, and CI secret scanning. Never carry legacy credentials/configuration into the rewrite.
  • RLS is deny-by-default. User-owned rows carry an owner identifier constrained to auth.uid(); the client cannot choose another owner. Storage paths and policies follow the same ownership rule. Realtime exposure is restricted by RLS and tested.
  • Minimize collected pantry data. Do not activate camera, AI, location, barcode, push, or payment collection without an approved purpose, disclosure/consent behavior where applicable, retention/deletion policy, provider review, and tests.
  • Define schema validation, canonical wire dates/quantities/IDs, migration reversibility or recovery, audit-safe error handling, and redacted observability before production data is accepted.

Migration posture

Do not migrate legacy Appwrite, SQLite, Firebase, or device data by assumption. First establish lawful ownership, source schema, data quality, consent, and rollback/recovery evidence. A migration increment must include mapping, dry-run, duplicate/malformed-record handling, reconciliation, rollback or compensating action, and user communication. I1 changes neither legacy data nor pam_pan/.

Delivery, CI, and release design

  • Pin JDK/Gradle/Android SDK toolchains in repository-visible configuration and CI. Record exact versions and commands with every increment gate. Do not select or assert Xcode/iOS toolchains in active CI.
  • CI builds and tests Android, runs Android contract/fixture checks, formatting/lint/static analysis, dependency and redacted secret scanning, documentation validation, and git diff --check. Cloud-dependent checks are explicit and isolated from offline foundation gates.
  • Release delivery uses the Android signing/distribution path only. Signing credentials and store credentials are never committed, printed, or embedded in public artifacts. Release promotion requires reproducible signed artifacts, provenance, test evidence, privacy/release notes, and rollback instructions.
  • Version contracts and migrations deliberately. Compatibility, rollback, incident ownership, telemetry privacy, retention, and alert thresholds must be established before production release.
