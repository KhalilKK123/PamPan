PAMPAN-NATIVE-REWRITE / I2A

Type: approved active increment plan · Authority: exact user-approved I2A scope · Feature: PAMPAN-NATIVE-REWRITE · Increment: I2A · Status: approved / active; implementation not yet verified · Planning base: 44a77789015cc19968bc609589a966ab0f6b9bf6 · Planning tree: b1fa6769a3d5847e6d3cee7b7717ef0681c08066 · Updated: 2026-07-23

Offline contract preview only. I2A defines a minimal pantry contract, validates fixed synthetic fixtures, and presents those fixtures in the Android application without persistence, authentication, network, cloud, mutation, background work, or relative-date behavior. It does not activate the remainder of I2 or I3–I9.

Objective and approved outcome

Establish the first narrow individual-pantry contract and a deterministic Android preview that can be verified entirely offline. A cold launch shows the heading “Pantry preview”, a prominent “Sample data — not saved” status, and exactly three fixed synthetic pantry items. Each item shows its name, decimal quantity with unit, exact expiry date, and category. The fixtures appear in their stable declared order only. A separately defined and tested empty state establishes the zero-item presentation.

The preview is sample-only evidence of the approved local contract. It is not stored user data, a migration result, a cloud schema, or a claim that CRUD, authentication, synchronization, or the later product workflow exists.

The schema, validator, fixtures, Kotlin model, UI mapping, and tests must express the same contract. Any extra field or semantic rule requires fresh approval.

Synthetic fixture and Android presentation

  • Define exactly three fixed, entirely synthetic pantry items and their synthetic categories. Do not derive them from production, personal, legacy, device, or service data.
  • Fixture order is stable declaration order solely to make the preview deterministic. It does not establish a general pantry ordering or sorting contract.
  • On offline cold launch, the Android screen shows “Pantry preview” as its heading and “Sample data — not saved” as a prominent, accessible status.
  • Each of the three fixture items exposes name, exact quantity and case-preserved unit, literal ISO expiry date, and resolved category name. Dates are not rewritten as relative prose.
  • Define and test the empty state using the same screen/model boundary with zero items. The empty state must not imply a loading, network, account, or persistence failure.
  • Keep the presentation usable with TalkBack and at 200% text without hiding, truncating beyond comprehension, or making the sample-only status ambiguous.

Planned implementation scope

  1. Add a versioned, machine-validated contract/schema and fixed synthetic fixtures under the existing reserved contracts/ boundary, plus a deterministic validator for the exact rules above.
  2. Add matching Kotlin domain/fixture mapping in apps/android/ without a new dependency, service, permission, network path, persistence layer, or lock change.
  3. Replace the foundation-only Android presentation with the approved offline pantry preview and defined empty state.
  4. Add focused contract, fixture, Kotlin model, UI, empty-state, accessibility, and cold-launch/connected tests needed to prove the approved behavior.
  5. Update only documentation that must describe the implemented and verified state after the implementation gates pass.

The implementation agent must determine the smallest repository-conforming file set. This plan does not authorize broad refactoring, dependency upgrades, build-system redesign, CI policy changes, or unrelated cleanup.

Acceptance gates

Contract and implementation consistency

  • The contract/schema/fixture validator accepts all three canonical fixtures and rejects representative violations of every required rule: blank IDs or names, malformed or calendar-invalid dates, negative or non-decimal quantities, non-string fixture quantities, blank or untrimmed units, and unresolved category IDs.
  • Kotlin models and parsing/mapping preserve opaque IDs, case-preserved units, exact decimal values, exact dates, and category-by-ID relationships without adding semantics.
  • Android unit/UI tests prove the exact heading, prominent sample-only status, three complete item presentations in fixture order, literal dates, and the defined empty state.
  • No new dependency, dependency-verification entry, lock selection, runtime permission, network permission, service, provider, or background component is introduced; the committed dependency and lock surfaces remain unchanged.

Local static, build, security, scope, and documentation gate

  • Run the existing contract validator and its negative test vectors.
  • Run the repository’s pinned Android formatting/static analysis, lint, unit tests, debug assembly, release-lock generation, dependency verification/lock no-drift, native and assembled audits, generated APK scan, and relevant UI-test build/preparation gates.
  • Run scanner self-tests and redacted repository/index/history/ref and generated-artifact security checks. Use synthetic fixtures only and publish no sensitive values.
  • Prove Android-only scope, dormant apps/ios/, unchanged reserved supabase/, and frozen pam_pan/ at 0512f84. Prove no excluded behavior or unrelated drift.
  • Run python3 scripts/update-doc-navigation.py, python3 scripts/check-docs.py, and the repository diff/scope checks.

Fresh exact-state physical-phone gate

Because I2A intentionally changes protected Android behavior, prior I1B evidence cannot carry forward. On the exact final implementation state, and only under a freshly approved physical-phone session, verify:

  • the approved connected Android tests pass on exactly one user-authorized wired phone;
  • with network connectivity disabled, a cold launch succeeds and presents the complete deterministic preview with no network, authentication, service, or persistence dependency;
  • TalkBack exposes the heading, sample-only status, item details, categories, and empty-state meaning in a coherent order without ambiguous unlabeled content;
  • at 200% text, both populated and empty states remain readable and operable without clipped essential information;
  • evidence is sanitized: retain no device serial, host fingerprint, private path, personal data, screenshot, or raw/broad log.

Independent final gates and authorized remote action

  • implement_verify and implement_review must both return PASS for the same exact implementation-state identifier after local and physical-phone evidence is complete.
  • Only after every pre-push gate passes, one non-force final branch push is authorized to produce one static-only GitHub Actions run at the exact final head.
  • The one run must pass all retained static analysis, unit, build/assembly, dependency-verification/lock, native-audit, APK/repository-security, documentation, and scope gates for that same commit/tree. Hosted emulator/system-image/AVD/ADB/connected/runtime/TalkBack work remains excluded and superseded in CI.

Explicit exclusions

  • No item or category create, edit, delete, or other mutation; no dependent-item deletion semantics.
  • No local persistence, cache, database, drafts, queued writes, sync, or conflict behavior.
  • No sorting, filtering, search, user-selectable ordering, unit conversion, quantity arithmetic, or near-expiry/expired calculation.
  • No dashboard, calendar, navigation expansion, reminder, notification, barcode, camera, AI, recipe, map, shelter, donation, payment, record, leaderboard, XP, widget, or other optional legacy feature.
  • No Supabase initialization, SDK, Auth, schema, migration, RLS, cloud project, Storage, Realtime, Edge Function, server contract, network call, or credential/configuration work.
  • No runtime permissions, signing, release, store, distribution, or legacy-data migration work.
  • No iOS implementation, build, test, verification, release, deletion, or rewrite. No pam_pan/ change.

Standing invariants and evidence limits

  • Android remains the only active native application. apps/ios/ remains dormant, pam_pan/ remains frozen at 0512f84, and supabase/ remains unchanged and reserved.
  • I1/I1E remains complete / PASS only for its historical exact state and revised static-only policy. I1C remains failed / BLOCKED historical evidence; I1D remains Diagnostic PASS only. I2A does not rewrite any prior result.
  • The repository is public. Fixtures and test vectors are synthetic. Reports retain categorical results, exact repository state, and safe digests only; they do not retain sensitive device identifiers, private host paths, personal data, raw logs, screenshots, or suspected values.
  • Static or source inspection cannot establish runtime, connected-test, TalkBack, or 200%-text acceptance. Those claims require the approved exact-state physical-phone gate.
  • I2 remainder and I3–I9 remain inactive. Any contract expansion, additional product behavior, cloud work, persistence, mutation, navigation, optional feature, or release activity requires fresh approval.

Stop conditions

  • Stop on contract ambiguity, schema/Kotlin/UI divergence, fixture data that is not demonstrably synthetic, dependency or lock drift, new permission/service/network behavior, or scope outside this plan.
  • Stop if the exact implementation state changes after phone evidence or either independent gate; repeat the affected gate only under fresh authorization where required.
  • Stop if the physical-phone gate cannot be performed safely without retaining identifiers or broad/raw logs. Do not substitute a hosted emulator or inference.
  • Stop on suspected disclosure without reproducing the value; follow the private escalation and provider revocation/rotation boundary.
  • Stop after the one authorized GitHub Actions run fails. No rerun, second push, force-push, amendment, alternate remote mutation, release, or expanded remediation is approved by I2A.

Completion rule

I2A remains approved / active until the contract, validator, fixtures, matching Kotlin implementation, populated and empty Android states, complete local gate, fresh exact-state physical-phone evidence, same-state implement_verify PASS, same-state implement_review PASS, one authorized static-only GitHub Actions run, and final documentation synchronization all pass. A partial result must be recorded as partial, failed, or blocked without activating the I2 remainder or I3–I9.
