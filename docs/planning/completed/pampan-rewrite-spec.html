4. Startup, authentication, navigation, and lifecycle

Observed startup sequence

  1. main() ensures bindings and awaits local-notification initialization.
  2. It attempts Firebase initialization and prints a caught failure.
  3. It calls Appwrite session deletion without awaiting, creates an email/password session using embedded demonstration credentials, then fetches the account.
  4. It mounts HomePage, or selected Appwrite failures mount ConnectionError.

Evidence: pam_pan/lib/main.dart:19-53. The embedded credential locations are main.dart:31-40 and dormant welcome/welcome_name.dart:73-81; values are not reproduced.

Observed defects/risks: network/auth gates UI; delete/create can race; notification initialization is uncaught; one handled branch can end without UI; no observed splash/auth observer/session restoration/router/dependency injection/central state.

9. Architecture, operations, risks, duplicates, and opportunities

  • Architecture/maintainability: UI directly performs auth, database, AI, permissions, files, scheduling, navigation; globals/circular imports, nullable dynamic contracts, samples mixed with live data, and stale imports/dependencies impede isolation.
  • Security: committed client configuration and embedded credentials; cleartext development endpoint; dead forms log passwords; unsafe release signing.
  • Privacy: package images may reach AI without observed minimization/disclosure; barcode/generated content can be logged; location/payment over-collect relative to proven features.
  • Reliability: unawaited mutations/false success/listener leaks/malformed dynamic records/missing retry-timeout/route races.
  • Accessibility: custom selectors lack explicit semantics; color-dependent selection; fixed layouts may overflow; carousel indicators unlabeled.
  • Performance: eager pantry work, unstable UUID keys, repeated account/image work, multiplying subscriptions, unbounded calendar accumulation.
  • Operations: no observed release pipeline, monitoring, incident ownership, migration tooling, environment strategy, platform CI, or validated signing/distribution.

Competing/dead implementations: auth (reachable Appwrite login, FrontPage, old login/signup, newer orphan signup); profile (Home drawer/orphan ProfilePage/old and placeholder editors); persistence (Appwrite/SQLite); calendar (Syncfusion/table_calendar); map (Google/OpenStreetMap); camera result (direct draft/commented image preview); widget (active ios/pampan/orphan template); and navigation (root/drawer/duplicated bottom bars). Other candidates: tips, XP, colors, FoodItemRow, five unused AI wrappers, getCustomCategories, unused assets, and seven unused direct dependencies.

Recommendations, not decisions: separate UI/application/integration boundaries; canonical typed models/errors; explicit startup/auth/onboarding/navigation states; select one implementation per feature; central validation; lifecycle-owned subscriptions; privacy controls; truthful mock labeling; deterministic parser/integration/lifecycle/accessibility/platform tests; dead dependency removal after authority decisions; and reproducible configuration/signing/monitoring/rotation controls.

10. Draft rewrite requirements and acceptance criteria

These are evidence-traceable rewrite requirements requested by approved I1 scope. They are draft planning content pending independent review; “must” is a proposed acceptance condition, not a claim about current behavior.

Functional requirements

  1. FR-01 Startup. Every initialization outcome must mount a tested visible retry/recovery state. Accept: injected success/auth/network/init failures each yield defined UI. Evidence: main.dart:19-53.
  2. FR-02 Authentication and ownership. Define signed-out/authenticating/authenticated/expired/unavailable states, await session mutations, document item/category ownership, and prove two-principal cross-owner denial. Accept: transition and authorization tests cover each state; no embedded credentials. Evidence: login.dart:25-55, home_page.dart:780-799, backend/appwrite_client.dart:15-194.
  3. FR-03 Onboarding, profile, and help. Decide whether onboarding/profile fields are in scope; if retained, make one authoritative flow reachable, persisted, resumable, and supported by accurate help/legal/privacy content; otherwise remove the orphan flows. Accept: the selected scope has completion/restart/logout/help tests and no placeholder claims. Evidence: welcome/, profile/help.dart:8-25.
  4. FR-04 Items and categories. Use typed create/edit drafts, a valid update ID, confirmed persistence before success, and one stable category ID/name/relationship contract. Accept: create/edit/failure recovery plus category create/rename/duplicate/delete/dependent-item tests. Evidence: add_item_page.dart:119-519, appwrite_client.dart:15-120.
  5. FR-05 Dashboard parity.
